## Escaped failure and root cause

The immutable `v0.1.0` tag passed Node 22 but intermittently failed the Node 20.11 GitHub Actions
job in `concurrent writers reclaim one stale queue lock without deleting a new owner`: zero of four
bounded reactions launched after the five-second lock timeout.

The failure was reproduced from the unmodified tag under Node 20.20.2. Several contenders could
validate the same stale directory before any rename. One contender moved it; another process then
created the replacement directory, and a delayed contender could move that new ownerless directory
before its `owner.json` write. The new owner write failed, and the old cleanup path could strand an
empty lock. The same pattern existed in the queue and state-store implementations.

## Structural correction

- Queue and state writes now use one dependency-free owner-claimed directory-lock primitive.
- Exactly one stale-reclaim claimant may proceed. The original stale directory remains pinned by
  an open file handle, and device/inode identity, owner token, and PID liveness are checked around
  the atomic rename. Holding the handle prevents rapid inode reuse from impersonating the original.
- Release renames only a token-verified owner to a unique tombstone before deletion. Owner-write
  races retry without an unvalidated recursive delete.
- A crashed reclaim claim is recoverable. Pre-claim directory timestamps are restored so recovery
  does not make the still-abandoned lock look fresh.
- Abandoned stale/release tombstones are pruned while live tombstone owners remain protected.
